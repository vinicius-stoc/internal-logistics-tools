ARQUITETURA DO PROJETO — PLATAFORMA INTERNA DE LOGÍSTICA

1. OBJETIVO DO SISTEMA

Este projeto consiste em uma plataforma web interna para acompanhamento gerencial de dados logísticos extraídos de uma planilha Excel disponibilizada pelo sistema interno da empresa via SFTP.

O sistema deverá:

* buscar ou receber uma planilha Excel de referência;
* tratar e validar os dados;
* armazenar os dados tratados no banco de dados;
* renderizar um dashboard gerencial;
* permitir filtros por período, motorista e pauta;
* permitir exportação da visão filtrada para Excel;
* controlar acesso por autenticação e permissões simples;
* permitir administração básica de usuários.

O projeto será desenvolvido como monolito Django, priorizando simplicidade, segurança, rastreabilidade e baixa complexidade operacional.

A regra principal da arquitetura é KISS: manter a solução simples, direta e adequada ao porte real do sistema.

2. CONTEXTO OPERACIONAL

Público-alvo:

* aproximadamente 5 funcionários da logística;
* uso interno;
* baixo tráfego;
* foco em visualização gerencial e redução de processos manuais.

Frequência de atualização:

* a importação dos dados ocorrerá estritamente uma vez por dia;
* não haverá atualização em tempo real;
* o dashboard sempre consumirá dados previamente gravados no banco.

Ambiente de desenvolvimento:

* todo o sistema deve poder rodar em localhost;
* a importação deve funcionar tanto com arquivo Excel local quanto com SFTP real;
* para protótipo e apresentação à diretoria, o sistema deve funcionar 100% localmente;
* a migração para PythonAnywhere pago ocorrerá somente após aprovação do custo.

Ambiente de produção previsto:

* PythonAnywhere pago;
* Django monolítico;
* banco de dados relacional;
* rotina diária agendada via Scheduled Task;
* SFTP real como fonte oficial da planilha.

3. DECISÕES ARQUITETURAIS APROVADAS

3.1. Padrão principal

O sistema será um monolito Django.

As variaveis e codigo deve seguir o padrão de nomes em ingles, nao quero variaveis em portugues-BR, manteremos em portugues-BR apenas o que estiver no front end visivel ao usuario

Não serão usados microsserviços, filas complexas, mensageria distribuída, Kafka, RabbitMQ, arquitetura orientada a eventos ou frontend SPA.

3.2. Frontend

O frontend será baseado em:

* Django Templates;
* HTML;
* Bootstrap;
* JavaScript simples;
* Chart.js;
* SweetAlert para mensagens visuais.

Não usar React, Vue, Angular ou outro framework SPA neste projeto, salvo decisão futura explícita.

3.3. Backend

O backend será baseado em:

* Django;
* Django Auth;
* Django Groups e Permissions;
* camada de services para regra de negócio;
* management commands para rotinas de importação;
* models simples e rastreáveis.

3.4. Banco de dados

Banco recomendado:

* PostgreSQL para ambiente definitivo;
* SQLite pode ser usado apenas em estágio inicial de protótipo local, se isso acelerar a validação visual;
* se possível, manter PostgreSQL também em localhost para reduzir diferença entre desenvolvimento e produção.

3.5. Hospedagem

Caminho recomendado:

* desenvolvimento e protótipo: localhost;
* produção após aprovação: PythonAnywhere pago.

Justificativa:

* baixo tráfego;
* poucos usuários;
* importação diária;
* necessidade de simplicidade operacional;
* PythonAnywhere atende bem Django + Scheduled Task;
* Render fica como alternativa futura caso o projeto evolua para deploy mais automatizado, workers persistentes ou banco gerenciado no mesmo provedor.

4. FORA DO ESCOPO

* microsserviços;
* Celery;
* Redis;
* Kafka;
* RabbitMQ;
* React;
* Vue;
* Angular;
* API REST completa;
* autenticação externa complexa;
* BI externo;
* data warehouse;
* cache distribuído;
* processamento em tempo real;
* importação acionada automaticamente por usuário comum;

7. MODELS APROVADOS

7.1. LeadTimeRecord

Model principal de negócio.

Representa uma linha tratada da planilha Excel.

Campos conceituais esperados:

* período;
* data de operação ou data de referência;
* motorista;
* pauta;
* origem, se existir na planilha;
* destino, se existir na planilha;
* status, se existir na planilha;
* lead time ou métrica equivalente;
* campos auxiliares extraídos da planilha;
* referência para ImportBatch;
* data de criação;
* data de atualização.

Observação:

A lista final de campos deve ser definida após análise real da planilha Excel de amostra. Nenhum campo deve ser inventado sem estar apoiado na planilha ou em necessidade validada.

7.2. ImportBatch

Model técnico obrigatório.

Representa uma execução da rotina de importação.

Campos conceituais esperados:

* nome do arquivo;
* origem da importação: local ou sftp;
* hash/checksum do arquivo;
* status;
* data/hora de início;
* data/hora de fim;
* total de linhas lidas;
* total de linhas válidas;
* total de linhas inválidas;
* mensagem de erro, se houver;
* detalhes técnicos resumidos, se necessário.

Status conceituais:

* processando;
* sucesso;
* erro;
* ignorado por duplicidade;
* arquivo não encontrado;
* erro de validação.

7.3. User

Usar o usuário padrão do Django, salvo necessidade explícita de customização.

Não criar um sistema próprio de autenticação.

7.4. UserProfile

Usar apenas se necessário para campos adicionais.

Responsabilidade principal:

* marcar se o usuário precisa trocar senha no primeiro acesso;
* armazenar metadados simples do usuário interno.

7.5. Group e Permission

Usar os recursos nativos do Django.

Grupos conceituais:

* LogisticaViewer;
* LogisticaAdmin;
* Superuser técnico.

Não criar tabela customizada de roles neste momento.

10. PIPELINE DE DADOS

10.1. Fluxo oficial

O fluxo de importação deve seguir esta sequência conceitual:

1. Scheduler ou desenvolvedor executa management command.
2. Command cria um ImportBatch com status processando.
3. Sistema identifica a origem da planilha: arquivo local ou SFTP.
4. Se origem for local, usa o caminho configurado.
5. Se origem for SFTP, conecta no servidor e baixa o arquivo para área temporária privada.
6. Sistema valida existência do arquivo.
7. Sistema calcula hash/checksum.
8. Sistema verifica se o arquivo já foi importado.
9. Sistema lê a planilha Excel.
10. Sistema valida colunas obrigatórias.
11. Sistema normaliza nomes de colunas.
12. Sistema trata tipos de dados.
13. Sistema valida linhas.
14. Sistema grava dados no banco.
15. Sistema atualiza ImportBatch com totais e status final.
16. Sistema remove arquivos temporários, quando houver.
17. Dashboard passa a consumir os dados atualizados do banco.

10.2. Regra principal

O dashboard nunca deve ler o Excel diretamente.

O dashboard nunca deve conectar no SFTP.

O dashboard sempre deve ler dados já tratados e persistidos no banco.

10.3. Idempotência

A importação deve evitar duplicidade.

Estratégias permitidas:

* hash/checksum do arquivo;
* nome do arquivo;
* data de referência;
* chave lógica por linha;
* combinação entre período, motorista, pauta e outros identificadores reais da planilha.

A chave final deve ser definida após análise da planilha.

10.4. Falhas esperadas

O sistema deve lidar com:

* SFTP indisponível;
* arquivo não encontrado;
* credencial inválida;
* planilha com layout alterado;
* coluna obrigatória ausente;
* linha inválida;
* arquivo duplicado;
* erro de banco;
* erro de permissão local.

Falhas na importação não devem derrubar o dashboard.

O dashboard deve continuar exibindo a última carga válida.

11. AMBIENTES

11.1. Localhost

O sistema deve rodar integralmente em localhost.

Recursos esperados:

* servidor Django local;
* banco local;
* planilha Excel de amostra;
* importação manual via management command;
* telas funcionais;
* autenticação funcional;
* dashboard funcional;
* exportação funcional, quando implementada.

No ambiente local, a origem preferencial da planilha será arquivo local.

11.2. Produção

O sistema será migrado para PythonAnywhere pago após aprovação.

Em produção:

* DEBUG deve estar desativado;
* SECRET_KEY deve ser segura;
* credenciais devem vir de variáveis de ambiente;
* SFTP será a origem oficial da planilha;
* importação será executada uma vez por dia via tarefa agendada;
* o sistema deve operar sobre banco persistente;
* arquivos temporários devem ser limpos após processamento.

13. SEGURANÇA

13.1. Credenciais

Credenciais nunca devem ser salvas no código.

Credenciais nunca devem ser commitadas.

Credenciais devem ser carregadas por variável de ambiente ou mecanismo equivalente.

Credenciais SFTP não devem aparecer em logs.

Credenciais de e-mail não devem aparecer em logs.

13.2. SFTP

Boas práticas:

* preferir chave SSH a senha, se possível;
* validar host/fingerprint, quando viável;
* usar usuário SFTP com permissão mínima;
* limitar acesso ao diretório necessário;
* não permitir escrita remota se leitura for suficiente;
* tratar timeout;
* registrar falhas de forma controlada.

13.3. Dashboard

Todas as rotas do dashboard devem exigir login.

Rotas administrativas devem exigir grupo/permissão adequada.

Exportação Excel deve exigir permissão.

Endpoints ou views de dados não devem ficar públicos.

13.4. Sessão e senha

Usar mecanismos nativos do Django.

Fluxos obrigatórios:

* login;
* logout;
* alteração de senha;
* troca obrigatória no primeiro acesso;
* reset de senha por e-mail.

* período;

* pauta.

15.2. Estratégia dos dados

O frontend não deve receber milhares de registros crus sem necessidade.

15.3. Chart.js

16. EXPORTAÇÃO EXCEL

A exportação deve seguir os mesmos filtros aplicados no dashboard.

A exportação deve ser implementada em service próprio.

A view deve apenas:

* validar login;
* validar permissão;
* capturar filtros;
* chamar service de exportação;
* retornar o arquivo.

A exportação não deve reprocessar a planilha original.

A exportação deve usar dados persistidos no banco.
